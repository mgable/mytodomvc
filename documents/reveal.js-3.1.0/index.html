What is End to End (E2E) testing

  • is not unit testing
  • tests a complete workflow
  • based on performing an action and testing the results
  • user centric
  • primary way to regression test

What is Protractor.js

  • built on Selenium / Webdriver
  • uses Jasmine syntax
  • based on asynchronous programming model

Developing a E2E testing strategy

  • determine aspects of behavior to test

Types of E2E testing

  • Rendering
  • Behavioral
  • Data
  • Performance

Render Tests

if a element rendered on the page

Behavior Tests

if the element behaves in the intended way

Data Tests

if the rendered data correct

Performance Tests

if the response times are within acceptable limits

Functional Specifications

When the submit button is pressed

  1. the name is required
  2. the name must have a minimum of two characters
  3. the name has a maximum of 256 characters
  4. the name can not have the following characters: \ / ? : *
  5. the name has to be unique within the system
  6. the name is submitted to the server

Functional Specifications interpreted

  1. the name is required
    • the submit button is disabled when the name field is empty
  2. the name must have a minimum of two characters
    • he submit button is disabled if the name is less than two characters
  3. he name has a maximum of 256 characters
    • the submit button is disabled if the name is more than 256 characters
  4. the name can not have the following characters: \ / ? : *
    • the submit button is disabled if it contains any of the restricted characters
  5. the name has to be unique within the system
    • enter a unique name
    • the submit button is disabled if the same unique name is entered again
  6. the name is submitted to the server
    • if a name passes all the above tests, the submit button is enabled

Developing a E2E testing strategy

  • determine aspects of behavior to test
  • write tests based on functional specifications
  • strive for an application which is always in an uniform starting state
  • write tests to make data when needed
  • write tests to stand alone
  • determine how much coverage is acceptable
  • determine gaps in coverage
  • determine what can not be tested

  1. enter a todo

    2. input field accepts user input
  2. save a todo
    1. places a new todo into the last index position of the todo list
    2. clears the input field
  3. edit a todo
    1. double-click a todo to edit
    2. ESC keypress reverts the input field to unmodified value
    3. RETURN keypress updates todo with input field text
  4. toggle the "complete" attribute of an individual todo
    1. mark an "active" todo as "complete"
    2. mark a "complete" todo as "active"

    1. mark all todos as "complete" if there is at least one "active" todo

    2. todo-count displays number of "active" todos
    3. a todo should be "active" when first created
    4. a todo should not "active" when marked "completed"
